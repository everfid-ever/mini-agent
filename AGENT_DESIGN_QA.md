# 架构设计题

每个模块选一题作答。选题原则:优先选我有一手实践的题;没有实践的模块,选能体现设计判断而不是知识背诵的题。

## 模块一:Context / Performance

**题目:一个 session 连续聊了 200 轮,context 快爆了。你会怎么做压缩?如何确保压缩后的对话仍然流畅?**

压缩的难点不在摘要本身的质量,而在两件更基础的事:触发判定的准确性,和压缩结果的生命周期管理。摘要质量主要取决于模型能力,后两者才是架构设计要解决的问题。

**触发时机。**在窗口占用达到一个阈值(如 80%)时主动触发,而不是等超窗后被动补救。留出余量的意义在于:触发压缩的那一轮,全量上下文仍然放得下——这样即使压缩失败,本轮对话仍可以用未压缩的全量继续,压缩失败只损失一次优化机会,不会中断服务。触发判定应基于真实的 usage token 数:上一轮模型响应返回的 prompt_tokens 是精确锚点,锚点之后新增的消息再做估算补齐。纯字符估算对不同语言、代码、多模态内容的偏差方向和幅度都不稳定,以它做触发判定,安全余量是不可靠的。

**压缩策略。**最近若干轮保留原文——指代、语气、正在讨论的细节,局部连贯性依赖这一段;更早的内容压成结构化摘要,槽位固定:任务目标、已确认的事实、关键决策及理由、未决事项、用户偏好。用固定模板而非自由摘要,是为了多次压缩之间结构稳定,后一次可以在前一次的基础上滚动更新,而不是对全部历史推倒重摘。两条硬约束:tool_call 与 tool_result 必须成对保留或成对丢弃,切开配对会导致后续请求不合法;system prompt 与任务目标不进入压缩范围。摘要注入回上下文时需带明确的身份标记,让模型将其识别为记忆回放而非用户的新输入。

**生命周期:压缩结果必须持久化。**这是最容易被忽略、后果也最严重的一环,我做过的一个对话 Agent 上实际发生过:压缩只改写了单次请求的内存上下文,而每轮从存储重建全量历史。于是跨过触发阈值后,每一轮都对越来越长的全量历史重新做一次摘要调用——首 token 延迟持续上升、压缩开销每轮重复发生;最终全量历史超过摘要模型自身的窗口,摘要必然失败,主模型同样超窗,会话进入不可恢复的报错状态。修复方式是把压缩边界(位置)和摘要内容一起落库,后续每轮重建上下文时从边界开始、头部注入摘要,使压缩成为跨轮滚动的增量过程。这次排查留给我一条一般化原则:多轮系统里,任何只存在于单次执行内存中的状态,都要回答"下一轮它还在吗"。

**失败处理。**摘要调用也是一次 LLM 调用,会抖动和限流。分两层:先做有限次数的指数退避重试,挡住瞬时故障;重试耗尽后不向上抛错,而是放弃本次压缩、退回全量上下文继续本轮——由于触发点留有余量,这个降级是安全的。压缩调用产生的 token 用量应与正常对话用量分开计量,否则无法评估压缩机制本身的净收益。

**流畅性的保证与验证。**用户感知的"压缩后变笨",除了对话记忆丢失,更常见的是运行时工作状态丢失——对话中途激活的能力、约定的输出格式、正在操作的对象,这些不在摘要模板覆盖范围内就会随原文一起消失,因此压缩流程中需要一个显式的状态结转步骤,把这类状态单独提取、原样带过压缩边界。验证用探针测试,且要可自动化:维护一组固定探针问题(指代早期实体、延续未决话题),压缩前后各跑一遍、自动比对回答一致性,纳入回归集;上线前把触发阈值临时调小,用短对话低成本复现"触发—压缩—后续轮次不再重复压缩"的完整链路。摘要模板中显式保留"未闭环的话题线",正是为了让这类探针可以通过。

## 模块二:Memory

**题目:和聊天 Agent 熟悉半个月后,用户问了一个以前问过的问题。Agent 如何做 memory 召回更合理?**

核心判断:召回质量的上限在写入侧,不在检索侧。多数方案在检索上投入(更好的 embedding、加 rerank),但如果检索对象是原始对话流,召回的永远是碎片——半个月的对话里,"那个问题"的完整语境散落在多轮消息中:提问、补充、结论各在不同位置。向量检索命中其中两三轮,拼不回当时的完整结论,模型只能基于碎片重新推理,答案很可能与上次不一致——不一致比"想不起来"更伤害信任。

**写入侧:从记对话到记结论。**在会话结束或消息量达到阈值时做一次固化,把情景记忆(聊了什么)提炼为语义记忆(确认了什么):问题、结论、结论成立的前提,每条带时间戳、话题标签和置信度。固化是一次离线 LLM 调用,发生在会话结束后的空闲时段,不占交互路径。检索对象是固化后的事实条目;原始对话流只作溯源存档。记忆同时必须是可变状态:新事实与旧条目冲突时更新或废弃旧条目,而不是 append-only 地堆积,把矛盾留给检索层。多窗口/多 session 并发时写入侧还有竞态:两个会话可能同时对同一事实固化出不同结论,需要用户粒度的单写者串行固化,或带版本号的 last-write-wins 并把冲突显式记录、留给下次固化消解。

**召回时机与延迟预算。**召回发生在交互热路径上,不能无条件对每条用户消息跑全套检索。合理的分层是:轻量的候选判定(关键词/embedding 近似匹配,毫秒级)对每条消息都做;判定有召回价值时才走完整的检索加重排。延迟上,召回可以与请求预处理并行执行,把开销藏进本来就存在的首 token 等待里;若某次检索超出预算,宁可本轮不注入,也不阻塞响应——记忆是增强,不是依赖。

**检索侧:混合召回,严格门控。**向量检索处理"同问异表"——这次的措辞和半个月前不同,语义相似度能兜住;关键词检索处理专有名词与精确指称——人名、产品名、代码符号,向量对这类 token 恰恰是钝的。两路并存,粗召回后过一层打分:相关性之外叠加时间衰减与重要性权重,被重复询问本身就是重要性信号。最重要的一条纪律是门控:宁可不注入,也不注入低置信度的记忆。错误记忆进入上下文是幻觉的直接来源——模型不会怀疑被喂入的"记忆",只会基于错误前提自信地推理。两种错误的代价不对称:漏召回的结果是模型承认记不清,错误注入的结果是一本正经的错误答案,因此阈值宁高勿低。

**使用侧:一致性加增量。**识别出重复提问后,正确的姿态不是机械复述上次的答案,而是:"上次问过这个问题,当时的结论是 A;这段时间发生了变化 B,因此现在修正为 C。"用户感受到的"熟悉感"来自 Agent 表现出记得、且在跟进,而不仅是答对。

**度量。**两个指标先行:重复提问识别率(用户重复问时成功召回历史问答的比例)衡量召回下限;错误注入率(注入记忆与当前问题无关或已过期的比例)衡量门控纪律。需要说明的是,这题里检索侧的判断有我做混合检索索引(模糊匹配与精确标签并存)的一手经验支撑,写入侧的抽取质量是推演——所以真正落地时我会先用这两个指标量出写入侧的真实水位,再谈检索参数优化。

## 模块三:Task

**题目:用户给 Agent 下达任务:每天早上 9 点根据昨天聊天情况做复盘总结。你会怎么设计?**

这道题的本质是离线任务系统,不是 prompt 工程。复盘模板是最简单的部分,真正的设计量在幂等、失败处理、投递保证,以及与在线会话的并发冲突。

**架构:调度与执行分离。**cron 调度器只负责在正确时间向任务队列投递一条任务消息,不执行业务逻辑;worker 从队列消费,完成取数、生成、落库、投递。理由是故障域隔离:执行侧故障不影响调度,次日照常触发;调度侧漏投,补投一条消息即可恢复。队列同时承担削峰与重试缓冲。

**幂等与状态分离。**去重键为 userID 加日期:调度重复触发、worker 崩溃重试、消息重复投递,都不会让用户收到两份同一天的复盘。生成状态与投递状态分开追踪:生成成功但推送失败,重推已生成的结果而不是重新生成——生成成本远高于推送,且重新生成的两份复盘措辞不同,用户会收到"两个版本的昨天"。

**失败路径要完整。**重试有上限,超限进入死信队列并告警,由人工或补偿任务处理——没有死信设计的重试等于无限循环。"昨天没有聊天"是正常的空输入而非失败,模板需处理空数据(跳过或轻量问候),不能让一个注定失败的任务每天早上进重试循环。投递侧需要定义"到达"的口径:推送通道的送达回执与应用内已读分别追踪,前者衡量系统,后者衡量产品。

**取数与生成。**按用户本地时区的昨日零点至今日零点取消息——"早上 9 点"天然是用户视角,时区按用户算,并处理夏令时切换。消息量大的用户依赖每日预聚合,复盘任务只读聚合结果,不在早高峰扫全量消息表。生成用固定模板(昨日主题、状态、产生的待办、今日建议),跨天结构稳定才可对比;单次生成设 token 上限。

**与在线会话的冲突。**后台任务运行时用户可能正在聊天,两者不能并发写同一份会话状态。方案是后台复盘使用独立的执行上下文,结果以独立消息投递进会话,不打断正在进行的交互;若系统已有会话级互斥(如每会话一把锁),后台任务排队等待而不是绕过。

**规模化与成本。**9 点是天然尖峰:按用户分片,把触发时间在一个小窗口内摊开;worker 并发度对齐 LLM 供应商限流配额,否则尖峰期大量限流错误反而拖垮成功率。成本上单任务设 token 上限、全局设日预算熔断——定时任务的费用乘以全量用户数,失控速度远快于在线对话。监控三条线:任务成功率、投递到达率、完成时刻延迟告警(超过预期窗口未完成的分片要报警),因为迟到的复盘和缺失的复盘对用户是同一种失望。

## 模块四:Tool / Session Runtime

**题目:如果 session state 为 busy,此时用户又发来新消息,或者异步工具完成事件也到达,runtime 应该如何处理?**

关键判断:这两类输入性质不同,不能用同一策略。异步工具完成事件是数据——唯一的要求是不丢;用户新消息带意图——可能是补充,也可能是对当前任务的否定,需要仲裁。把两者塞进同一个"busy 即拒绝"的逻辑,是这道题最常见的错误设计。

**完成事件:永不拒绝,安全点消费。**完成事件进入 session 的事件队列(mailbox),正在运行的 run 在下一个安全点——模型步骤的边界——消费它;session 空闲则直接触发恢复执行。拒绝完成事件意味着任务结果永久丢失。配套设计:超时和失败同样建模为完成事件,走同一条消费路径,这样"任务超时"能被模型看到并向用户解释,而不是所有环节一起无限等待。

**用户消息:按意图分流。**补充类(在当前任务上追加要求)排队等本轮结束,或在安全点注入当前 run;否定类(推翻当前方向)应取消当前 run,再以新消息开启新 run;意图无法判定时默认排队,并在界面明示"将在当前任务完成后处理"。底线是:最差的设计是硬拒绝且不提供取消入口——用户面对一个已经跑偏的任务只能等它烧完。

**取消的语义。**取消不是 kill,是一个有收尾语义的操作,四件事缺一不可:停在步骤边界、已生成的部分输出持久化、用量计量完整、向前端发显式终止帧。硬 kill 的两个后果:撕裂进行中的 tool_call/tool_result 配对,历史中留下无结果的调用,后续请求不合法,会话从"忙"变成"坏";用户已经看到的流式内容不落库,刷新后凭空消失,比等待更困惑。这套语义我在一个多实例部署的 Agent runtime 上完整落过地:每个会话一把分布式锁保证单写者,busy 时新消息收到明确的"会话忙"错误,但配有完整的取消链路——取消信号经消息队列广播覆盖所有实例(run 可能跑在任意一台上),发起实例本地取消作为零延迟主路径,广播兜底跨实例;安全点停止、部分输出落库、终止帧收尾之后,用户的实际体验是"拒绝但可一键取消重发",而不是死等。

**安全点粒度的缺口要正视。**安全点是模型步骤边界,意味着一个长耗时的工具调用本身不可在中途打断——取消也要等它执行到边界。补法有三层:工具实现层面传递取消信号(如透传 context cancellation),让支持协作式取消的工具提前退出;不支持的工具给取消操作设一个宽限期,超过则放弃等待、接受该工具调用的结果作废;产品层面向用户明示"正在停止,等待当前操作结束",把等待变成有预期的状态。

**理想形态。**每个 session 一个逻辑单写者(actor):所有输入——用户消息、完成事件、取消信号——统一进入其队列,由 session 自己的循环在安全点串行仲裁。队列内部需要优先级:取消信号插到最前(它的价值随延迟衰减最快),完成事件次之,普通消息按序。并发在入口处被消解,业务代码中不再需要分散的状态检查与加锁;分布式部署下,单写者由会话粒度的锁或一致性哈希路由保证。业界框架已开始内置这一形态:eino v0.9 的 TurnLoop 就是现成实现——mailbox 缓冲全部输入、Push 时可按安全点抢占(安全点显式枚举为"模型调用后/工具调用后",可配抢占超时)、配合 checkpoint store 自动区分"turn 中途被打断"与"turn 之间退出"两种恢复。所以从更保守的"锁加拒绝"起步,演进路径是清晰且低成本的:锁升级为队列,拒绝变为排队,取消信号通道复用为消息注入通道——保守解允许存在,但要确认它在通往理想解的路径上。

## 模块五:Agent Runtime 架构对比

**题目:OpenHands 的状态机设计有什么优缺点?更优雅的实现方式是怎么样的?**

先描述对象。OpenHands 的核心架构是事件流加显式状态机:agent 的动作(Action)与环境反馈(Observation)进入统一事件流;AgentController 维护一组显式状态(RUNNING、AWAITING_USER_INPUT、PAUSED、STOPPED、ERROR、FINISHED 等),集中驱动状态转移。

**优点是实在的。**状态显式,系统当前所处位置一目了然,可观测、可断言、易调试;事件流承担事实记录,历史可重放、会话可恢复、审计接近免费;agent 决策逻辑与沙箱 runtime 经事件流解耦,替换任一侧不影响另一侧。对开源项目还有工程之外的价值:新贡献者看一眼状态枚举即可建立系统的心智模型。

**结构性问题:两个事实源。**事件流已经是事实源,又独立维护一份显式状态,系统里就有了两份需要保持同步的真相。代价体现在两处。其一,状态枚举随功能膨胀:加 delegation、加 pause 语义都要加状态,而转移逻辑是状态与事件的笛卡尔积,分散各处难以穷举——"等待用户输入时异步事件到达"这类交叉情形,漏写一个分支就是难以复现的线上异常,测试也很难覆盖全组合。其二,两个事实源可能失一致:状态字段已是 FINISHED,事件流末尾却还有未消费的 observation——这类缺陷没有单点修复,因为错的不是某行代码,而是"状态需要被同步维护"这个前提。模块四那类并发输入问题,正是这个结构性问题最集中的爆发点。

**更优雅的方式:事件溯源加派生状态。**事件日志作唯一事实源;"当前状态"不独立存储,而是事件日志的投影函数,需要时折叠(fold)得出。状态由此永远不会与历史不一致——它就是历史的函数;新增功能变成"增加事件类型、修改投影函数",而不是修改一张不断膨胀的转移矩阵;配合单写者串行消费(每个 session 一个消费循环),状态乘事件的组合爆炸收敛为一个循环体内的顺序判断。我把自己负责的 Agent 系统重构成过这个形状:领域层只产出领域事件,不做副作用;应用层消费同一份事件流,分别完成流式推送、状态更新、持久化——一份事实,多个下游各自投影。实践下来,许多状态可以直接派生而无需字段:能力激活态从消息窗口派生,忙闲从锁的存在性派生,run 结束由终止事件表达,系统中不存在中央转移表。再往上一层是 durable execution(Temporal 一类):控制流直接写成代码,框架经事件溯源自动持久化与恢复执行位置,"等待输入"退化为一行 await;代价是重型依赖与确定性执行约束。对本身就需要事件流做回放的 Agent 场景,事件溯源加派生状态这一层是收益成本比最好的落点。

**这个方案自己的代价也要说清。**投影有计算成本,规模上去后需要物化视图缓存;"系统现在处于什么状态"不再一眼可见,排查时要折叠事件流,调试体验不如直接读状态字段;事件 schema 的版本演进是真实的生产难题——事件语义变更后,旧事件如何被新投影正确解释,需要版本化事件与投影的升级策略,这笔账在选型时就要算进去;可观测性需要靠把事件模型设计得足够细(每个关键动作都是独立事件)来补回,事件流设计粗了,派生不出有用的状态,两头落空。

**适用边界。**对小型、状态少、单实例的 Agent,显式状态机是务实的正确解——几个状态、十几条转移,一张表看得清清楚楚,引入事件溯源属于过度工程。事件溯源的优势在状态与事件的组合开始爆炸、并发输入需要仲裁、历史需要回放的中大型系统上才成立。所以更准确的结论不是"事件溯源更好",而是:这两种方案的差别,是把设计功夫花在维护状态转移表上,还是花在设计事件模型上——系统复杂度越过某个点之后,后者的边际成本更低。
